MDN: Validating forms

  • Client side form validation-making sure all required form controls are filled out in the correct format. Validation is done in the browser and is called client-side. Validation done on the server is called server-side validation.
    • Types of client-side validation
    1. Built in form validation using HTML5 validation features. Built in form validation has better performance than javascript but is not as customizable.
    2. Javascript validation is completely customizable but you have to creat eit all or use a library.
  • If something on a form is invalid it blocks the user from sending in the form.
  • Pattern is another useful validation feature. A regular expression or regex is a pattern that can be used to match character combinations in text strings.

MDN: Using fetch

  • The fetch API provides JS an interface for accessing and manipulating parts of the http pipeline such as requests and responses.
  • fetch() won't reject on HTTP error status but will resolve normally and only reject on network failure or if something prevents the request from completing.
  • fetch() won't send cross-orgin cookies.
  • fetch('http://example.com/movies.json')
    .then(response => response.json())
    .then(data => console.log(data));
  • fetch can be used to send a request with credentials, upload JSON data, upload a file, uploading multiple files, to process a text file line by line, checking that fetch was successful, etc.
  • The headers interface allows us to create our own headers object via the Headers() constructor. A headers object is a simple multi-map of names to values.
    • Possible Guard Values:
    • none: default
    • request: guard for a headers object obtained from a request
    • request-no-cors: guard for a headers object obtained from a request created with Request.mode no-cors.
    • response: guard for a headers object obtained from a response (Response.headers).
    • immutable: guard that renders a headers object read-only; mostly used for ServiceWorkers.
  • Response instances are returned when fetch() promises are resolved.
    • The most common response properties you will use:
    • Response.status: An integer (default value 200) containing the response status code.
    • Response.statusText: A string (default value ""), which corresponds to the HTTP status code message. Note that HTTP/2 does not support status messages.
    • Response.ok: seen in use above, this is a shorthand for checking that status is in the range 200-299 inclusive. This returns a Boolean.
